You can use a Worker as a step in your Pipeline. Worker transformations are useful to filter, or enrich, incoming data. Worker transformations run in the middle of a Pipeline: after records are ingested, but before records are written out to the destination.

Worker transformations run asynchronously, and scale out horizontally to keep up with load. Pipelines handle serializing and de-serializing data, so transformations can operate on objects. Finally, since these transformations run as Workers in your account, you can configure access to services such as [D1](/d1/) or [KV](/kv/).

## Tutorial
To use Worker transformations in your Pipeline, you need to:
1. Create a new Worker
1. Define transformations
1. Deploy the Worker
1. Create a Pipeline which includes the transformation
1. Send data, and verify that transformations are running
1. Verify output in R2

### 1. Create a new Worker
Create a Cloudflare Worker if you don't already have one. This Worker will be used to transform inbound records.

To create a Worker, run the following command in a terminal:

<PackageManagers
	type="create"
	pkg="cloudflare@latest"
	args={"pipeline-transformation"}
/>

<Render
	file="c3-post-run-steps"
	product="workers"
	params={{
		category: "hello-world",
		type: "Hello World Worker",
		lang: "TypeScript",
	}}
/>

This will create a new directory, which will include both a `src/index.ts` Worker script, and a [`wrangler.toml`](/workers/wrangler/configuration/) configuration file. Navigate into the newly created directory:

```sh
cd pipeline-transformation
```

### 2. Define transformations
You will now define transformations inside your Worker. From your Worker project directory, open the `src` folder using your preferred code editor. Add the following to the `index.ts` file:

```ts
import { PipelineBatchMetadata } from 'cloudflare:pipelines'
// @ts-ignore - just for now :)
import { PipelineTransformationEntrypoint } from 'cloudflare:pipeline-transform'

// Define the expected type of raw records that the Pipeline will receive
type InputType = {
	foo: string
}

// Define the expected type of records after they are transformed
type OutputType = {;
	foo: string
	modified: boolean
	pipelineId: string
}

// Pipelines will invoke the run() method defined in this class
export class MyTransform extends PipelineTransformationEntrypoint {
	async run(
	records: InputType[],
	metadata: PipelineBatchMetadata
	): Promise<OutputType[]> {
		try {
			console.log('Working on batch from pipeline: ',metadata.pipelineName)
			return records.map((record) => {
				const mutated = record as OutputType;
				mutated.modified = true
				mutated.pipelineId = metadata.pipelineId
				return mutated
			})
		} catch (err) {
			console.error('Error in transformation: ', err)
			throw err
		}
  }
}

export default {
	async fetch(request, env, ctx): Promise<Response> {
		return new Response('Hello World!');
	},
} satisfies ExportedHandler<Env>;
```

Transformations are defined by extending the `PipelineTransformationEntrypoint` class. The transformation definition must define a `run()` method, which contains the primary logic for the transformation. As seen above, the `run()` method receives objects from a Pipeline, and returns objects back to the Pipeline. The transformation above adds two fields to each incoming record.

A single Worker can contain multiple transformation definitions, as long as each has a unique class name. You can also leverage resources such as KV, or D1, using [bindings](/workers/runtime-apis/bindings).

### 3. Deploy your worker
With your `index.ts` file configured, you are ready to publish your worker. To publish your Worker, open up the terminal window that we used to create our Worker in step 2. Run the following command:

```sh
npx wrangler deploy
```

You should see output that resembles the below:

```
Uploaded <YOUR-WORKER-NAME> (0.76 sec)
Published <YOUR-WORKER-NAME> (0.29 sec)
  https://<YOUR-WORKER-NAME>.<YOUR-ACCOUNT>.workers.dev
```

### 4. Create a pipeline which includes the transformation
You are now ready to create a new Pipeline which includes the transformation. If this is your first time using Pipelines, go through the instructions in the [get started guide](/pipelines/get-started) to familiarize yourself.

Include a transformation in your Pipeline by adding the following flag when creating a Pipeline: `--transform-worker <worker-name>.<entrypoint>`. For example:
```sh
npx wrangler pipelines create [PIPELINE-NAME] --r2-bucket [R2-BUCKET-NAME] --transform-worker pipeline-transformation.MyTransform
```

You should see output that resembles the below:
```sh output
🌀 Authorizing R2 bucket "[R2-BUCKET-NAME]"
🌀 Creating pipeline named "[PIPELINE-NAME]"
✅ Successfully created pipeline [PIPELINE-NAME] with ID [PIPELINE-ID]

You can now send data to your pipeline with:
  curl "https://<PIPELINE-ID>.pipelines.cloudflare.com/" -d '[{ "foo":"bar }]'
```

### 5. Send data, and verify that transformations are running
You are now ready to send data to your Pipeline.

You can get a live tail of logs from your transformation worker by running [`wrangler tail`](/workers/wrangler/commands#tail). Open the terminal window we used to create our Worker in step 1, and run `wrangler tail` to get live logs.

You should see output which looks like this:
```sh
Successfully created tail, expires at 2025-03-14T22:08:04Z
Connected to pipeline-transformation, waiting for logs...
```

Now, open a new terminal window. Run the curl command you received in step 4 to post sample data to your Pipeline:
```sh
curl "https://<PIPELINE-ID>.pipelines.cloudflare.com/" -d '[{ "foo":"bar }]'
```

You should see new logs appear from your Worker:
```sh
Successfully created tail, expires at 2025-03-14T22:08:04Z
Connected to pipeline-transformation, waiting for logs...
(log) Working on batch from pipeline:  <pipeline-name>
```

### 6. Verify output in R2
Go to the R2 bucket you created in step 2 via [the Cloudflare dashboard](https://dash.cloudflare.com/). You should see a prefix for today's date. Click through, and you'll find one or more files, containing the records you sent in step 5. You'll see that the records have been enriched according to the logic defined in step 2, with two additional fields: `mutated` and `pipelineId`.

By following this tutorial, you've created a Pipeline with a Worker transformation. To use this in a real setting, modify the code samples above to suit your logic.

## Batch settings
Transformations run on batches of records. Batching is useful to reduce the number of times your Worker is invoked, thus reducing costs. To specify batch settings for your Pipeline, refer to the [batching](/pipelines/configuration/batching) documentation.

## Bindings (`env`)
You can use [bindings](/workers/runtime-apis/bindings) from inside your transformation worker. Bindings allow your transformation Worker to interact with resources, such as [D1](/d1/), [KV](/kv/) or environment variables. The `env` object is exposed as a class property of the `PipelineTransformationEntrypoint`.

For example, you can declare a binding to the environment variable GREETING:
<WranglerConfig>

```toml
name = "my-transformation-worker"

[vars]
GREETING = "Hello"
```

</WranglerConfig>

You can access it by calling `this.env.GREETING`:

```js
import { PipelineTransformationEntrypoint } from 'cloudflare:pipeline-transform'

export class MyTransform extends PipelineTransformationEntrypoint {
	async run(
	records,
	metadata
	) {
			return records.map((record) => {
				const mutated = record;
				mutated.greeting = this.env.GREETING
			})
  }
}
```

You can use any type of [binding](/workers/runtime-apis/bindings) this way.

## Lifecycle methods (`ctx`)
The [`ctx`](/workers/runtime-apis/context) object is exposed as a class property of the `WorkerEntrypoint` class.

For example, you can extend the lifetime of the invocation context by calling the `waitUntil()` method:

```js
import { PipelineTransformationEntrypoint } from 'cloudflare:pipeline-transform'

export class MyTransform extends PipelineTransformationEntrypoint {
	async run(
	records,
	metadata,
	) {
			return records.map((record) => {
				const mutated = record;
				this.ctx.waitUntil(this.doSomething())
			})
  }

	async doSomething() {
		// ...
	}
}
```
